# Effective timestamps: switch storage from two columns to tstzrange

## Problem

Entities declared with `withEffectiveTimestamps()` currently generate two nullable
columns: `effective_from timestamp(3) with time zone NULL` and
`effective_to timestamp(3) with time zone NULL`. We want to switch to a single
Postgres `tstzrange` column, per the pattern in `many-to-many-history.md`.

## Scope

- Applies to entities using `withEffectiveTimestamps()` (both
  `hasSingleEffectiveRecord = true` and `= false`).
- DB column + DAO/row-mapper query generation only. The Kotlin entity class,
  DTOs, services, and Angular frontend are unaffected — they continue to use
  `effectiveFrom: Instant?` / `effectiveTo: Instant?` exactly as today.
- `withEffectiveLocalDates()` (daterange equivalent) is out of scope.
- GiST exclusion constraints / overlap-prevention indexes from
  `many-to-many-history.md` are deferred — not generated by this change.
- Flyway migration reconciliation for `maia-showcase` is out of scope for the
  generator change; the user will reconcile the regenerated DDL with the
  Flyway migration scripts manually.

## Data model

Replace:

```sql
effective_from timestamp(3) with time zone NULL,
effective_to   timestamp(3) with time zone NULL
```

with:

```sql
effective_range tstzrange NOT NULL DEFAULT tstzrange(now(), NULL)
```

Notes:

- Default bound type `'[)'` (inclusive lower, exclusive upper) preserves the
  existing semantics for non-null bounds: `effective_range @> :ts` is
  equivalent to `effective_from <= :ts AND (effective_to > :ts OR effective_to
  IS NULL)`.
- `lower(effective_range)` / `upper(effective_range)` return `NULL` for
  unbounded ends, and `tstzrange(NULL, x)` / `tstzrange(x, NULL)` create
  unbounded bounds — so `effectiveFrom`/`effectiveTo` being `null` round-trips
  cleanly through the range column without the column itself ever being NULL.
  Note this is a deliberate semantic change for `effectiveFrom = null`: an
  unbounded lower bound means "effective since the beginning of time", so
  `effective_range @> :ts` is now `true` for such rows (previously, `NULL <=
  :ts` was `NULL`/falsy, excluding the row).
- The Kotlin entity class fields (`effectiveFrom: Instant?`, `effectiveTo: Instant?`)
  do not change. The entity row mapper and DAO are responsible for translating
  between the single DB column and the two entity fields.

## Generator changes

### 1. DDL generation (`CreateTableSqlRenderer`)

For entities with `hasEffectiveTimestamps = true`, special-case the
`effectiveFrom`/`effectiveTo` field pair: skip generic per-field column
rendering for these two fields and instead emit one column:

```sql
effective_range tstzrange not null default tstzrange(now(), null)
```

### 2. Query generation (`JdbcDaoRenderer` / `RowMapperRenderer`)

Introduce shared helper(s) (e.g. on `EntityDef` or a renderer util) that,
given an entity's field list, collapse the `effectiveFrom`/`effectiveTo` pair
(when `hasEffectiveTimestamps`) into:

- **Select projection**: for all `select *` query sites (DAO finders, bulk
  queries, the many-to-many join-fetch query), append
  `, lower(effective_range) as effective_from, upper(effective_range) as effective_to`
  (using the appropriate table alias, e.g. `mtm.effective_range` for the join
  table in the many-to-many join-fetch query). `RowMapperRenderer`'s existing
  `rsa.readInstantOrNull("effective_from"/"effective_to")` calls require no
  change.

- **Insert / bulk-insert / insertHistory**: collapse the two columns/value
  placeholders into one column `effective_range` with value expression
  `tstzrange(:effectiveFrom, :effectiveTo)`. `SqlParams.addValue(Instant?)`
  binding for `effectiveFrom`/`effectiveTo` is unchanged (still two
  `timestamptz` params).

- **Update / setFields**: collapse the two assignments into
  `effective_range = tstzrange(:effectiveFrom, :effectiveTo)`.

- **`findEffectiveBy*` WHERE clause**: replace
  ```sql
  and effective_from <= current_timestamp
  and (effective_to > current_timestamp or effective_to is null)
  ```
  with
  ```sql
  and effective_range @> current_timestamp
  ```

### Affected call sites (JdbcDaoRenderer)

All locations currently emitting `select * from ...` for entities with
`hasEffectiveTimestamps`, plus:

- `renderFunctionInsertFor` (insert, insertSubclass, insertHistory, bulk
  insert via `input_rows`)
- update/`setFields` column-assignment generation
- `findEffectiveBy*` where-clause generation
- `RowMapperRenderer`'s many-to-many join-fetch query
  (`mtm.effective_from, mtm.effective_to` → `lower(mtm.effective_range) as effective_from, upper(mtm.effective_range) as effective_to`)
