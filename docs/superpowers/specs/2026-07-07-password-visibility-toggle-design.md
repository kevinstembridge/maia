# Password Visibility Toggle — Design

## Goal
Password input fields (`HtmlInputType.password`) generated by the Angular form renderers should include a show/hide toggle, automatically, with no spec changes required.

## Background
`HtmlInputType.password` is defined in `maia-gen-spec/.../HtmlInputType.kt`. It's currently only used by `maia-showcase`'s login form. Regardless of which builder sets it (`AngularFormDefBuilder`, `EntityCreateHtmlFormDefBuilder`, `RequestDtoHtmlFormDefBuilder`, or the `EntityCreateApiDef`/`EntityUpdateApiDef` construction sites), it always ends up on an `AngularFormFieldDef`, and rendering always goes through `MatFormFieldRenderer`. The other two field-def types that also expose `htmlInputType` (`HtmlFormFieldDef`, `HtmlFormEntityFieldDef`) are not consumed by any `<input>`-rendering code path today — they only feed enum-constant generation in `AppModuleGenerator`. So this feature only needs to touch `AngularFormFieldDef`-based rendering.

There are two rendering paths, both in `MatFormFieldRenderer.kt`:
- `renderHtmlInputField` — signal-form path (`[formField]="dialogForm.x"` binding), used by `AbstractCrudFormHtmlRenderer` (and callers of `renderSignalFormField`).
- `renderReactiveFormHtmlInputField` — reactive/entity-form path (`formControlName="x"` + `FormGroup`), used by `FormHtmlRenderer`, `EntityEditReactiveFormHtmlRenderer`, `AbstractCrudReactiveFormHtmlRenderer`.

Both currently emit `type="${htmlFormField.htmlInputType}"` as a static attribute with no branching on the enum value anywhere in the codebase.

Existing precedent for a suffix toggle control inside `mat-form-field`: `mat-datepicker-toggle matIconSuffix` / `mat-timepicker-toggle matSuffix` in `renderReactiveFormDatepickerField`. Existing precedent for a per-field boolean signal on the component class: `<field>IsVisible = signal<boolean>(false)` (linked fields, `AngularSignalFormComponentRenderer.kt`) and `<field>ShowFormSignalName = signal(false)` (timestamped fields, `AngularReactiveFormComponentRenderer.kt`). `signal` (from `@angular/core`) and `MatButtonModule` are already unconditionally imported by both component renderers; `MatIconModule` is not imported by default in either.

## Behavior
Applies automatically to every field with `htmlInputType == HtmlInputType.password`, in both the signal-form and reactive-form (incl. entity-form) rendering paths. No new DSL method.

### Template changes (`MatFormFieldRenderer.kt`)
In both `renderHtmlInputField` and `renderReactiveFormHtmlInputField`, when the field's `htmlInputType == HtmlInputType.password`:

- Replace the static `type="password"` attribute with a binding:
  `[type]="<classFieldName>PasswordVisible() ? 'text' : 'password'"`
- Immediately after the `<input .../>` self-close, emit a suffix toggle button inside the same `<mat-form-field>`:
  ```html
  <button mat-icon-button matSuffix type="button" (click)="<classFieldName>PasswordVisible.set(!<classFieldName>PasswordVisible())" [attr.aria-label]="<classFieldName>PasswordVisible() ? 'Hide password' : 'Show password'">
      <mat-icon>{{ <classFieldName>PasswordVisible() ? 'visibility_off' : 'visibility' }}</mat-icon>
  </button>
  ```
  (`type="button"` prevents accidental form submission.)

For non-password fields, behavior is unchanged (static `type="..."` attribute, no button).

### Component class changes
In both `AngularReactiveFormComponentRenderer.kt` and `AngularSignalFormComponentRenderer.kt`:

- Add a new class-fields step (alongside the existing per-field-group steps) that, for each field in `angularFormDef.htmlFormFields` where `htmlInputType == HtmlInputType.password`, emits:
  `    <classFieldName>PasswordVisible = signal(false);`
- In the imports step, add `MatIconModule` (`@angular/material/icon`) conditionally, when `angularFormDef.htmlFormFields.any { it.htmlInputType == HtmlInputType.password }`.
- Use `htmlFormFields` (not `formModelFields`) as the source list for both of the above, since that's the exact list `FormHtmlRenderer`/`AbstractCrudFormHtmlRenderer`/etc. iterate over to emit the `<input>` elements — guarantees the generated signal always matches what the template references.

### Naming
Signal name: `<classFieldName>PasswordVisible`. Chosen to avoid collision with the existing `<classFieldName>IsVisible` signal (used for conditionally-shown linked fields).

## Out of scope
- No opt-out/opt-in flag — this is default behavior for all password fields.
- No change to `HtmlFormFieldDef`/`HtmlFormEntityFieldDef` — they don't drive input rendering today.
- No change to password validation, strength meters, or autocomplete attributes.

- Run the showcase app and manually confirm the toggle switches the input between masked/plain text and the icon/aria-label update accordingly.
- Confirm existing generator tests (if any target `MatFormFieldRenderer` or the component renderers) still pass, and non-password fields are unaffected.
