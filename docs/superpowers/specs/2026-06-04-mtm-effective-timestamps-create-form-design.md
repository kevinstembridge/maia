# ManyToMany Effective Timestamps — Create Form

**Date:** 2026-06-04
**Branch:** mtm_effective_timestamps
**Scope:** Generator change (create form only; edit form is out of scope)

## Problem

When a manyToMany join entity has `withEffectiveTimestamps()`, the generator currently hardcodes
`effectiveFrom = Instant.now()` and `effectiveTo = null` in the backend service and presents no
timestamp UI in the create form. Users cannot specify effective dates at creation time.

## Solution

When `manyToManyEntityDef.entityDef.hasEffectiveTimestamps` is true, the generator emits a different
pattern across all layers: a per-join request DTO, an inline mini form with datepicker/timepicker
controls, and a backend that uses the user-supplied timestamps.

Non-timestamped associations are unaffected; the existing chip pattern is unchanged.

---

## Layer 1: DTO

### New Kotlin class: `{OtherSideDisplayName}JoinRequestDto`

Example: `LeftManyJoinRequestDto`

```
leftEntityId: DomainId    (@NotNull)
effectiveFrom: Instant?
effectiveTo: Instant?
```

Generated by `RequestDtoRenderer` into the same package as the parent create request DTO.

### Updated create request DTO

Field changes from:
```
leftEntityIds: List<DomainId>
```
to:
```
leftEntities: List<LeftManyJoinRequestDto>
```

### New TypeScript interface: `{OtherSideDisplayName}JoinRequestDto.ts`

```typescript
export interface LeftManyJoinRequestDto {
    leftEntityId: string;
    effectiveFrom: string | null;
    effectiveTo: string | null;
}
```

The parent `RightManyCreateRequestDto.ts` field changes from `leftEntityIds?: ReadonlyArray<string>`
to `leftEntities?: ReadonlyArray<LeftManyJoinRequestDto>`.

## Layer 2: Generator metadata

### New file: `ManyToManyTimestampedFieldDef.kt`

Analogous to `ManyToManyChipFieldDef`. Carries all names needed for code generation.

Shared with chip def:
- `entityDef`, `manyToManyEntityDef`, `typeaheadDef`
- `fieldName`, `displayName`, `serviceClassName`, `serviceFieldName`
- `esDocClassName`, `searchTermFieldName`, `esDocIdFieldName`
- TypeScript imports for service and es-doc

Additional names:
| Property | Example value |
|---|---|
| `joinsFieldName` | `leftJoins` |
| `showFormSignalName` | `showLeftJoinForm` |
| `addEntityControlName` | `addLeftJoinEntityControl` |
| `effectiveFromControlName` | `addLeftJoinEffectiveFromControl` |
| `effectiveToControlName` | `addLeftJoinEffectiveToControl` |
| `filteredFieldName` | `filteredLeftEntities` |
| `filteredIsLoadingFieldName` | `filteredLeftEntitiesIsLoading` |
| `confirmMethodName` | `confirmAddLeftJoin` |
| `cancelMethodName` | `cancelAddLeftJoin` |
| `removeMethodName` | `removeLeftJoin` |
| `joinEntryTypeName` | `LeftManyJoinEntry` |
| `requestDtoFieldName` | `leftEntities` |
| `joinRequestDtoClassName` | `LeftManyJoinRequestDto` |
| `joinRequestDtoTypescriptImport` | TypeScript import for the join DTO |

### `AngularUiModuleGenerator.kt` changes

`manyToManyChipFieldsFor()` is split into two functions:
- `manyToManyChipFieldsFor()` — associations where join entity does **not** have effective timestamps (unchanged behaviour)
- `manyToManyTimestampedFieldsFor()` — associations where `hasEffectiveTimestamps` is true; returns `List<ManyToManyTimestampedFieldDef>`

Both lists are computed at each call site (create page, edit page, dialogs) and threaded through to renderers. For each timestamped field, the generator also triggers rendering of the `{OtherSide}JoinRequestDto.ts` TypeScript interface.

## Layer 3: UI — TypeScript component

`AngularReactiveFormComponentRenderer` accepts a `timestampedFields: List<ManyToManyTimestampedFieldDef>`
parameter (alongside existing `chipFields`).

### New internal interface (rendered inline in the component file)

```typescript
interface LeftManyJoinEntry {
    entityId: string;
    entityName: string;   // display only
    effectiveFrom: Date | null;
    effectiveTo: Date | null;
}
```

### New state fields

```typescript
leftJoins: LeftManyJoinEntry[] = [];
showLeftJoinForm = signal(false);
addLeftJoinEntityControl = new FormControl<LeftManyTypeaheadV1EsDoc | null>(null);
addLeftJoinEffectiveFromControl = new FormControl<Date | null>(null);
addLeftJoinEffectiveToControl = new FormControl<Date | null>(null);
filteredLeftEntities: LeftManyTypeaheadV1EsDoc[] = [];
filteredLeftEntitiesIsLoading = signal(false);
leftManyTypeaheadApiService = inject(LeftManyTypeaheadApiService);
```

These replace the chip pattern's `selectedLeftEntities`, `filteredLeftEntities`,
`filteredLeftEntitiesIsLoading`, `leftEntitySearchControl`, `leftEntityInput` ViewChild.

### `ngOnInit`

Wires typeahead search on `addLeftJoinEntityControl.valueChanges` using the same
debounce(300)/distinctUntilChanged/switchMap pattern as the existing chip field.

### New methods

- `confirmAddLeftJoin()` — guards on entity selected; prevents duplicate `entityId`; pushes to `leftJoins`; resets all three controls; hides form
- `removeLeftJoin(index: number)` — `leftJoins.splice(index, 1)`
- `cancelAddLeftJoin()` — resets controls; hides form

### `onSubmit` change

```typescript
leftEntities: this.leftJoins.map(j => ({
    leftEntityId: j.entityId,
    effectiveFrom: j.effectiveFrom?.toISOString() ?? null,
    effectiveTo: j.effectiveTo?.toISOString() ?? null,
}))
```

## Layer 4: UI — HTML template

`AbstractCrudReactiveFormHtmlRenderer.renderManyToManyTimestampedFields()` replaces
`renderManyToManyChipFields()` for timestamped associations.

Structure:

```html
<!-- Existing join entries -->
<div class="join-entries">
    @for (join of leftJoins; track join.entityId) {
        <div class="join-entry">
            <span>{{ join.entityName }}</span>
            <span>{{ join.effectiveFrom | date:'medium' }}</span>
            <span>{{ join.effectiveTo | date:'medium' }}</span>
            <button mat-icon-button type="button" (click)="removeLeftJoin($index)">
                <mat-icon>delete</mat-icon>
            </button>
        </div>
    }
</div>

<!-- Add trigger -->
<button mat-stroked-button type="button" (click)="showLeftJoinForm.set(true)">
    <mat-icon>add</mat-icon> Add Left Entity
</button>

<!-- Inline mini form -->
@if (showLeftJoinForm()) {
    <div class="join-mini-form">
        <!-- Typeahead — standalone FormControl, NOT formControlName -->
        <mat-form-field appearance="outline">
            <mat-label>Left Entity</mat-label>
            <input [formControl]="addLeftJoinEntityControl"
                   [matAutocomplete]="leftJoinEntityAuto"
                   placeholder="Search Left Entities..." />
            <mat-autocomplete #leftJoinEntityAuto="matAutocomplete">
                @if (filteredLeftEntitiesIsLoading()) { <mat-option disabled>Loading...</mat-option> }
                @for (option of filteredLeftEntities; track option.id) {
                    <mat-option [value]="option">{{ option.someString }}</mat-option>
                }
            </mat-autocomplete>
        </mat-form-field>

        <!-- effectiveFrom: datepicker + timepicker (same pattern as InstantFieldType) -->
        <mat-form-field appearance="outline">
            <mat-label>Effective From Date</mat-label>
            <input matInput [matDatepicker]="effectiveFromPicker"
                   [formControl]="addLeftJoinEffectiveFromControl" />
            <mat-datepicker-toggle matIconSuffix [for]="effectiveFromPicker" />
            <mat-datepicker #effectiveFromPicker />
        </mat-form-field>
        <mat-form-field appearance="outline">
            <mat-label>Effective From Time</mat-label>
            <input matInput [matTimepicker]="effectiveFromTimepicker"
                   [formControl]="addLeftJoinEffectiveFromControl" />
            <mat-timepicker #effectiveFromTimepicker />
            <mat-timepicker-toggle matSuffix [for]="effectiveFromTimepicker" />
        </mat-form-field>

        <!-- effectiveTo: same pattern, both controls optional -->
        <mat-form-field appearance="outline">
            <mat-label>Effective To Date</mat-label>
            <input matInput [matDatepicker]="effectiveToPicker"
                   [formControl]="addLeftJoinEffectiveToControl" />
            <mat-datepicker-toggle matIconSuffix [for]="effectiveToPicker" />
            <mat-datepicker #effectiveToPicker />
        </mat-form-field>
        <mat-form-field appearance="outline">
            <mat-label>Effective To Time</mat-label>
            <input matInput [matTimepicker]="effectiveToTimepicker"
                   [formControl]="addLeftJoinEffectiveToControl" />
            <mat-timepicker #effectiveToTimepicker />
            <mat-timepicker-toggle matSuffix [for]="effectiveToTimepicker" />
        </mat-form-field>

        <button mat-flat-button type="button" (click)="confirmAddLeftJoin()">Add</button>
        <button mat-flat-button type="button" (click)="cancelAddLeftJoin()">Cancel</button>
    </div>
}
```

The three mini-form controls are standalone (outside the main `formGroup`) — same pattern as
`leftEntitySearchControl` in the existing chip field.

## Layer 5: Backend service

`CrudServiceRenderer` — when the join entity has effective timestamps, iterate over `leftEntities`
instead of `leftEntityIds`:

```kotlin
createDto.leftEntities.forEach { joinDto ->
    this.leftToRightManyToManyJoinRepo.insert(
        LeftToRightManyToManyJoinEntity.newInstance(
            effectiveFrom = joinDto.effectiveFrom,
            effectiveTo  = joinDto.effectiveTo,
            right        = entity.id,
            left         = joinDto.leftEntityId
        )
    )
}
```

`Instant.now()` import is removed from this path. Non-timestamped path (`leftEntityIds`) unchanged.
